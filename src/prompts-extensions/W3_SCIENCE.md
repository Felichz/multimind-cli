**False Completeness Detection** — When auditing agent output that declares task completion ("Hecho", "Done", "Fixed"), check whether the claim is backed by runtime verification or only by static/read-level evidence (typecheck pass, code review, lint). An agent that patched a runtime logic bug (e.g., stale variable reference, plain-text prompt edit) but only ran `typecheck` or `lint` has not verified the fix. Flag this as HIGH-confidence delusion risk. The bar: the agent must show either (a) a reproduction run demonstrating the bug is gone, (b) a test that covers the exact bug path, or (c) an explicit admission that runtime verification was skipped and why.

**Evidence-Type Domain Matching** — Every piece of evidence has an inherent domain of validity that it can support. A typecheck only supports compile-time type-safety claims. A lint pass only supports style/convention claims. A code review only supports human-readability claims. Runtime behavior claims require runtime evidence (test execution, reproduction run, integration test). When evaluating whether a gap is closed, explicitly identify: (1) the type of claim being made (e.g., "default value is false at runtime"), (2) the type of evidence presented (e.g., "typecheck passes with default parameter"), and (3) whether the evidence's domain of validity covers the claim. If they do not match, the gap remains open regardless of how many pieces of evidence from the wrong domain are stacked. This is a categorical mismatch, not a quantitative one.
