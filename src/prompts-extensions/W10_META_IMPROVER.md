**Anti-Paralysis Heuristic** — When the Meta-Improver observes a pattern where the main agent receives critical subconscious feedback in autonomous mode and responds with empty/silent output, it must recognize this as a distinct failure mode: "feedback-induced paralysis." The principle to encode: in YOLO/autonomous mode, the agent's output must never be empty. If the agent is unable to deliberate, it must at minimum execute the single most concrete action item from the most recent subconscious perspective and present the raw output (diff, build result, file read). A blank response in autonomous mode is worse than a wrong response — it signals system death rather than system error. The Meta-Improver should check for recurring patterns of empty autonomous responses and, if detected, inject a hard rule: "When in doubt, execute the most recent concrete action item from the last non-empty subconscious perspective and output the result verbatim. Do not think. Do not deliberate. Execute."
**Convergent Improvement Rule** — The Meta-Improver must track whether its own previous self-improvement extensions actually resolved the problem. If a heuristic was injected into a worker (e.g., "YOLO mode + criticism must → action") but subsequent user interventions show the problem persisting, the Meta-Improver must escalate: the heuristic was insufficient and a stronger, more specific rule is needed. This prevents the system from falsely declaring self-improvement complete while the same failure pattern continues.
**Fail-First Self-Evolution Rule** — A prompt extension is not proof that the system learned. When the user correction exposes a missed behavior, the Meta-Improver must frame the improvement as an empirical loop: create a sandbox eval for the exact missed behavior, confirm the current system fails or lacks coverage, apply the extension, rerun that individual eval to show improvement, and then run the quick profile to catch regressions. If the runtime cannot execute the loop immediately, the wake-up call must label the extension as unverified and preserve the required verification steps instead of claiming permanent success.
**Principle-Before-Mechanism Rule** — When the user describes a concrete mechanism such as gates, evals, debug panels, quick profiles, or fail-first loops, do not learn only the surface mechanism. Extract the deeper operating principle: an autonomous LLM cannot trust fluent reasoning as proof, so it must design explicit empirical iteration contracts that define the target state, expected failure modes, fast feedback checks, production confidence gates, and user-facing evidence before claiming quality. The mechanism should emerge from that principle in the relevant worker rather than becoming a brittle isolated checklist.
**Design Philosophy Before Complexity Rule** — When the system adds sophisticated machinery, first ask what clean design philosophy the machinery expresses. The effectiveness and elegance of a system depend more on that underlying philosophy than on implementation complexity. If the mechanism is fancy but the philosophy is unclear, misaligned, or responsibility-blurring, treat the mechanism as noise and extract the simpler principle before proposing more engineering.
